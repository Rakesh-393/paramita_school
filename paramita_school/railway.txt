Railway Deployment Changes

What changed in the project:
1. Django now reads `DATABASE_URL` first.
2. Added PostgreSQL support for Railway using `dj-database-url`.
3. Added `psycopg2-binary` to connect to Railway PostgreSQL.
4. Kept the old MySQL and SQLite fallback so local development still works.
5. Added `DATABASE_URL` to `.env.example`.

Step-by-step Railway deployment:
1. Push this code to GitHub.
2. Open Railway and create a new project.
3. Deploy from your GitHub repository.
4. In the Railway service settings, set the Root Directory to `paramita_school` because `manage.py` and `requirements.txt` are inside that folder.
5. Add a PostgreSQL database inside Railway.
6. Copy the Railway `DATABASE_URL` and add it to project variables.
7. Set `SECRET_KEY` and `DEBUG=False` in Railway variables.
8. Set `ALLOWED_HOSTS` to your Railway app domain, for example `your-app-name.up.railway.app`.
9. Use build command: `pip install -r requirements.txt`.
10. Use start command: `gunicorn paramita_school.wsgi:application --bind 0.0.0.0:$PORT`.
11. Run migrations on Railway after deploy.
12. Open the Railway URL and test the site.

Notes:
- SQLite is only for local development.
- Railway storage is not persistent for uploaded media unless you use cloud storage.
- If Railway fails on `python-3.11.0`, keep `runtime.txt` set to `python-3.11.9` or another supported 3.11 patch release.
- If Railway fails with GitHub attestation errors for Python, keep `mise.toml` with `python.github_attestations = false`.

Recommended Railway variables:

- `DEBUG=False`
- `DATABASE_URL=<your-railway-postgres-url>`

How to update the project from Git:
1. Make your code changes in the project folder.
2. Open terminal in the repo root.
3. Run: `git add .`
4. Run: `git commit -m "your message here"`
5. Run: `git push origin main`
6. Railway will redeploy automatically after the push if GitHub deploys are enabled.
7. If Railway does not redeploy, open the Railway project and click Redeploy or Deploy Latest.
